Интеграционная архитектура «Эллинга»

Как продукт стыкуется с реальным ландшафтом заказчика: 1С трёх поколений, Битрикс24 в облаке и коробке, легаси-почта, существующие базы знаний, ЕИС и ГИСП. Принцип: интегрируемся по протоколам, а не по плагинам, и работаем слоями деградации — продукт полезен даже клиенту, у которого есть только почта.

00 · Статус

Что из этого уже построено, а что пока текст

Документ описывал замысел. Ниже — что после реализации работает и проверено тестами, а что осталось намерением. Разделение важнее самой архитектуры: на завод нельзя приносить слайд под видом продукта.

Проверка на живой 1С заказчикане проводилась только мок — это главный неснятый риск слоя
Дыра, которую нашла собственная проверка. Принцип «агент внутри контура сам ходит наружу» выполнен только в коробке. Сейчас приём вызывается из процесса сервера: в облаке до 192.168.x завода маршрута нет физически, и кнопка «Забрать заказы» работать не может — таймаутом это не лечится. Обещанной отдельной службы рядом с 1С в коде нет. Граница при этом проведена верно: модуль не зависит от сервера ни одной строкой, поэтому вынос в отдельный процесс — работа на день. Но пока в облаке живут только папка обмена на общем диске и Мост. На демо это незаметно: мок-1С поднимается на 127.0.0.1, куда сервер дотягивается.
Четыре ошибки, найденные замером, а не рассуждением. (1) Сущности 1С называются по-русски — Document_ЗаказПокупателя; без кодирования пути коннектор не заработал бы ни на одном заводе, оставаясь зелёным на латинских именах. (2) 1С чаще висит, чем падает: тесты били по «соединение отвергнуто» (мгновенно), а зависание нашлось только на самодемо — человек ждал 20 секунд. (3) Кириллический хост ронял HTTP-клиент с ordinal not in range(256) — лечится punycode. (4) Суммы 1С приходят как «27 500 000,00»; «27.5 млн» в русском заказе читается как опечатка.

01 · 1С

Три поколения — три механизма

Ключевое решение — «Эллинг Агент»: лёгкая служба Windows, которая ставится в локальную сеть клиента рядом с 1С. Внутрь она говорит с 1С родными механизмами (OLE, OData, файлы), наружу — с нашим сервером по HTTPS (в закрытом контуре — с локальной инсталляцией). Так мы не требуем от клиента открывать 1С в интернет и не зависим от версии его конфигурации.

Поколение у клиентаМеханизмФакты (валидировано)Наш выбор
1С 6.0 / 7.5 и DOS-учёткина многих предприятиях старше 7.7 прямое чтение DBF-таблиц файловой базы (read-only); регулярная выгрузка CSV/DBF силами клиента OLE/API отсутствуют; базы этих поколений — открытые DBF-файлы, структуры известны и читаются любым XBase-драйвером; писать в них извне нельзя (риск разрушения базы) Агент: read-only DBF-коннектор по расписанию, только чтение; запись — никогда; фолбэк — мастер импорта Excel/CSV. Если и этого нет — уровень 0 матрицы деградации (почта + Excel) остаётся полноценным входом

1С 8.2/8.3 + УППснята с поддержки: полное прекращение 01.04.2027 OData (платформа ≥ 8.3.5), HTTP-сервисы (≥ 8.3.5), файловый XML/планы обмена OData — автоматический REST без доработки конфигурации: чтение справочников/документов; ограничения: нет отчётов, слабые $expand, деградация на больших выборках → только инкрементальные малые выборки OData на чтение + наша обработка .epf для событий; конфигурация «заморожена» — интеграция переживёт снятие с поддержки, т.к. платформа обновляется
1С:ERP / УТ 11 / БП 3современные HTTP-сервисы (кастомный REST), EnterpriseData, OData HTTP-сервисы — полный контроль контракта; типовая практика интеграций CRM↔1С HTTP-сервисы — готовое расширение «Эллинг-Обмен», ставится без снятия с поддержки

Контракт данных v1 (минимальный и достаточный)

  • Из 1С к нам: контрагенты, договоры (+ИГК и признак ГОЗ), счета и их оплата, отгрузки/реализации. Инкрементально, раз в 15 минут (агент) или файлом раз в час (7.7-фолбэк).
  • От нас в 1С: ничего в фазе 1 (принцип: не трогаем учёт — раздельный учёт ГОЗ живёт в 1С). Фаза 2: черновики заказов покупателя.
  • Идемпотентность по GUID 1С, очередь с retry, журнал обмена виден администратору клиента.

02 · Битрикс24

Два издания — два способа

Издание у клиентаКак ставимсяТехнические рамки (валидировано в 1-й волне)
Облако Битрикс24Тиражное приложение «Эллинг Sync» в Маркете (нужен статус техпартнёра 1С-Битрикс; у клиента должна быть подписка Маркета — без неё REST не работает вообще)Лимиты leaky bucket: 2 req/s, всплеск до 50 (Энтерпрайз 5/250); batch до 50 команд; офлайн-события вместо push — переживают недоступность приёмника; обход деградации list-выборок через start=-1 + фильтр по ID; retry на 503/429
Коробка (в т.ч. закрытый контур)Локальное приложение (client_id/secret или вебхуки) — Маркет в коробке без интернета недоступен, поэтому поставляем коннектор вместе с инсталляцией Эллинга, классическая лицензияREST-лимиты в коробке настраивает администратор; полный доступ к D7 не используем — остаёмся на REST, чтобы не зависеть от доработок клиентского портала

Сценарий данных: сделки/компании/контакты ↔ заказы Эллинга (двусторонне), задачи — односторонне к нам (производство живёт у нас), виджет-плейсмент «статус заказа» в карточке сделки Б24. Конфликт-резолюция: последняя запись побеждает, лог конфликтов админу.

03 · Почта и календарь

Легаси-почта — главный интеграционный канал MVP

  • Приём: IMAP (IDLE для мгновенных входящих) — работает с чем угодно, включая почту на своём сервере. Адрес заказа (zakaz-2417@…) + BCC-копирование + пересылка.
  • Exchange 2010/2013 on-prem: протокол EWS жив — Microsoft выводит EWS только в Exchange Online (окт. 2026 — апр. 2027), «no changes to EWS in Exchange Server». Аутентификация Basic/NTLM. Через EWS читаем и отправляем, календарь — приглашениями.
  • Отправка: SMTP клиента (письма уходят с домена клиента — важно для доверия верфей). Сроки — ICS-приглашения, работают в любом Outlook.
  • MAPI не трогаем — закрытый, тяжёлый, не нужен при живых IMAP/EWS.

05 · Государственные источники

ЕИС и ГИСП: без парсинга HTML

  • ЕИС (тендеры): анонимный FTP закрыт с 01.01.2025 — теперь официальный канал это СОИ ЕИС (SOAP/XML, нужен токен/ЭП) + раздел «Открытые данные». Решение MVP: партнёрский API агрегатора (Мультитендер/DaMIA — платные API существуют; Тендерплан/Контур берут данные из выгрузок ЕИС + 370 ЭТП) — быстрее и дешевле, чем самим вставать на СОИ. Собственная интеграция СОИ — фаза 3, когда объём оправдает.
  • ГИСП / реестр ПП 719: публичный веб-каталог есть (gisp.gov.ru/pp719v2/pub), официального документированного API нет. Решение: изолированный модуль-адаптер (внутренние JSON-эндпоинты каталога, обновление раз в сутки, деградация в «проверь вручную по ссылке» при изменении сайта). Риск нестабильности принят и локализован — это и есть причина, почему фичу не делает никто из больших: им дорого поддерживать, нам это ров.

Пять выводов: (1) агент в LAN снимает и проблему версий 1С, и страх «открывать учёт в интернет»; (2) учёт не трогаем — мы статусный слой над 1С, а не замена; (3) для коробочных клиентов всё ставится офлайн — ни один канал не требует облака; (4) госисточники берём через агрегаторов, пока объём не оправдает прямые каналы; (5) EWS, OLE и read-only DBF — «скучные» технологии, но именно они открывают 80% реального парка заказчиков, включая учётки старше 1С 7.7; (6) в древние базы никогда не пишем — только читаем: целостность учёта клиента дороже любой фичи.
